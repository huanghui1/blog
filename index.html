前端动画优化

发表于 2017-03-23

提高动画的优化不得不提及浏览器是如何渲染一个页面。在从服务器中拿到数据后，浏览器会先做解析三类东西：

  • 解析html,xhtml,svg这三类文档，形成dom树。
  • 解析css，产生css rule tree。
  • 解析js，js会通过api来操作dom tree和css rule tree。

解析完成之后，浏览器引擎会通过dom tree和css rule tree来构建rendering tree：

  • rendering tree和dom tree并不完全相同，例如： 或 display:none 的东西就不会放在渲染树中。
  • css rule tree主要是完成匹配，并把css rule附加给rendering tree的每个element。

在渲染树构建完成后，

  • 浏览器会对这些元素进行定位和布局，这一步也叫做reflow或者layout。
  • 浏览器绘制这些元素的样式，颜色，背景，大小及边框等，这一步也叫做repaint。
  • 然后浏览器会将各层的信息发送给GPU，GPU会将各层合成；显示在屏幕上。

如上所说，渲染树构建完成后；浏览器要做的步骤：
reflow——》repaint——》composite

明白浏览器的渲染原理，我们尝试来对动画进行一些优化：

  • 避免浏览器reflow和repaint
  • 优化GPU动画

避免浏览器reflow和repaint

reflow和repaint都是耗费浏览器性能的操作，这两者尤以reflow为甚；因为每次reflow，浏览器都要重新计算每个元素的形状和位置。
下面是一些针对reflow和repaint的最佳实践：

  • js代码中不要获取元素的宽高值，应将其缓存

  • 不要一条一条地修改dom的样式，尽量使用className一次修改。

  • 将dom离线后修改
    使用documentFragment对象在内存里操作dom。
    先把dom节点display:none;（会触发一次reflow）。然后做大量的修改后，再把它显示出来。
    clone一个dom节点在内存里，修改之后；与在线的节点相替换。

  • 不要使用table布局，一个小改动会造成整个table的重新布局。

  • transform和opacity只会引起合成，不会引起布局和重绘。

优化GPU动画

只用GPU动画

现代浏览器都会用GPU参与绘制，我们所要做的就是限制CPU参与动画绘制。

在这个例子中，对于动画的每一帧；浏览器会计算元素的几何形状，渲染新状态的图像，并把它们发送给GPU。
如果CPU在执行耗时的js代码，动画会有一些卡顿，要知道，js线程和UI线程是互斥的。
那如果我们使用transform代替left动画呢？

浏览器在动画执行之前就知道动画如何开始和结束，因为浏览器没有看到需要reflow和repaint的操作；浏览器就会画两张图像作为复合层，一张是A图像，一张是B图像和整个页面。并将他们的动画参数一起传入GPU。动画的每一帧都不需要CPU参与。
GPU是在一个独立的线程中绘制图像的，所以即便js代码在执行，也丝毫不会影响动画。

我们看到这里重要的是复合层，它是高效绘制的关键，将需要动画的元素提升为一个layer，即便需要repaint，也只会repaint这个layer，不会repaint整个页面。
以下是浏览器会提升一个元素为layer的条件：

  • 3d或透视变换css属性，例如translate3d,translateZ等等（js一般通过这种方式，使元素获得复合层）
  • 混合插件（如flash）。
  • 元素自身的 opacity和transform 做 CSS 动画。
  • 拥有css过滤器的元素。
  • 使用will-change属性。
  • position:fixed。
  • 元素有一个 z-index 较低且包含一个复合层的兄弟元素(换句话说就是该元素在复合层上面渲染)

内存优化

使用GPU动画需要发送多张渲染层的图像给GPU，GPU也需要缓存它们以便于后续动画的使用。

一个渲染层，需要多少内存占用？为了便于理解，举一个简单的例子；一个宽、高都是300px的纯色图像需要多少内存？

300 300 4 = 360000字节，即360kb。这里乘以4是因为，每个像素需要四个字节计算机内存来描述。

假设我们做一个轮播图组件，轮播图有10张图片；为了实现图片间平滑过渡的交互；为每个图像添加了will-change:transform。这将提升图像为复合层，它将多需要19mb的空间。800 600 4 * 10 = 1920000。

仅仅是一个轮播图组件就需要19m的额外空间！

而且，还记得上面列举的提升一个元素为layer的条件，其中有一点就是元素有一个 z-index 较低且包含一个复合层的兄弟元素。居一个例子就是上面的AB两个元素运动，如果B发生动画并提升为一个layer，即便A元素上面也不做，浏览器也会将它提升为一个layer，因为它符合上面列举的最后一条，元素有一个 z-index 较低且包含一个复合层的兄弟元素。这就是隐式合成。隐式合成增加了layer数量，进而增大了内存占用
以下列举了GPU绘制优化的tip：

  • 避免浏览器的隐式合成。
  • 减小复合层的尺寸5%——10%，然后使用scale将它们放大。

用karma+mocha+chai测试前端代码

发表于 2017-03-21

测试框架

测试框架采用karma+mocha+chai,mocha是测试框架,chai是断言库,提供很多功能强大且不同风格(BDD和TDD)的断言,并且有许多插件可以扩展chai的功能,例如sinon-chai. karma是测试运行器,它可以将我们的测试运行在不同的浏览器(如chrome, PhantomJS),并且可以实时监控我们的测试和js文件,一旦有改动,就会重新跑测试.最最重要的是,karma的社区很活跃,有很多解决方案.
Atomus是一个不错的前端测试工具,我们的js代码有着大量的DOM操作, 当我们进行单元测试时，我们意识到我们所做的是DOM模拟。在大多数情形下我们对UI不感兴趣，而对它的行为感兴趣。Atomus则引入了jsdom,在node环境下提供了DOM操作,而且还很方便让我们对请求进行mock.看起来很完美,但是对于我们代码用requireJS进行模块化, karma有了专门的解决方案karma-requirejs, Atomus只有一个参考,并没有解决方案.

写测试前的准备

让代码变得可测试的.

这里有一篇参考.因为单元测试注重的是测试函数,因此可以将函数分为两类,一类是工具函数,就是将许多的方法公用的地方抽取出来,只提供一类单一的功能.这类函数有比较容易测试,只要有确定的输入,测试预期的输出即可.这一类的函数如果有太多的mock,就需要考虑重构了.另一类是处理业务逻辑的函数,这一类并不能测试预期的输出,有可能是操作了DOM,有可能是以特定的参数调用了另一个函数.这些都视情况而定.

编写测试用例

如何编写测试

使用chai BDD风格的API

若是chai的API不能满足需求,可以到这里寻找chai插件

Mock

因为测试的确定性,确定的输入产生预期的输出.然而我们的代码常常会有依赖,所以我们需要mock确保输入是确定的.

  • XHR mock:
    可以使用sinonjs的server API

  • 方法mock:
    可以使用sinonjs的stubs API

Spy:检测函数调用,可以使用sinonjs的spies-api,也可以使用chai-spies 这个插件

测试HTML

为何测试HTML,有两个原因:

  • Datapanel的代码里有大量的DOM操作,为了确保这些操作时正常
    在datapanel页面里,因为业务需要,常常需要获取当前的统计内容是什么,就写了一个通用的方法getCategories来获取,对这个函数进行测试成功后,有可能这个函数还是没有达到预期的效果,因为这个方法是根据页面上单元框的选中的内容获取的,假设这个单元框不存在,那这个方法就会获取不到统计内容.
  • 即便是已经记住要加上需要操作的标签,有可能也会犯错:
    比如一个绑定click时间的元素,用jquery选择器是这样选择的,$(‘’.dataPanelMiddle .main-menu’’),当HTML重构的时候,应该不会忘记加上class= main-menu的元素,但很有可能会忘记把这个元素嵌套在class= dataPanelMiddle的元素内,因此jquery选择不到这个元素,事件也会添加失败.

如何测试HTML

综上所述,只需要测试HTML是否存在这个DOM元素就可以了.
首先需要引入HTML文件,文件只会在karma的服务器上,并不会在页面加载,需要我们请求获取到文件.可以使用karma-html2js-preprocessor插件,安装配置好了以后,只需要简单的加载就可以使用了.
